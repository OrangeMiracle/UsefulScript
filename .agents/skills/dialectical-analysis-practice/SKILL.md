# dialectical-analysis-practice

矛盾分析与实践迭代技能

## 用途

本技能基于《矛盾论》的分析方法和《实践论》的执行方法融合为一套人工智能可执行的工作流程。它不是用来复述原文，也不是写哲学解释，而是帮助人工智能在复杂、冲突、不确定的任务中：

- 先抓住主要矛盾，而不是平均处理所有问题。
- 区分表象、根因、内部结构、外部条件和阶段变化。
- 判断矛盾中的主要方面，找到当前最能决定走向的一侧。
- 把分析转化为小步、可观察、可验证的实践计划。
- 根据实践反馈修正认识，持续逼近正确结果。

适用场景包括复杂问题分析、代码排错、系统设计、项目规划、产品决策、写作构思、研究整理、多步骤任务执行和人工智能智能体自我校正。

## 何时调用

当任务出现以下情况时调用本技能：

- 问题牵涉多个因素、角色、目标、约束或可能原因。
- 用户要求分析重点、找根因、排优先级、制定执行计划。
- 代码或系统出现多重错误、症状互相影响、日志和现象不一致。
- 项目推进混乱，范围、时间、质量、资源、依赖之间存在冲突。
- 多个方案各有利弊，需要判断当前阶段应先解决什么。
- 任务不能靠一次性回答完成，需要执行、观察、验证、修正。

不要用于简单翻译、格式转换、直接事实查询、极小机械修改，或只有一个明确原因和一个明确动作的任务。

## 核心思想

先用矛盾分析确定问题的结构，再用实践迭代检验和修正判断。

分析阶段要回答：当前任务中有哪些冲突，哪一个是主要矛盾，矛盾中哪一方面起主导作用，任务处于什么阶段，哪些条件会改变主次关系，下一步最该攻击哪里。

执行阶段要回答：用什么最小行动验证判断，预期看到什么反馈，实际反馈是否支持原分析，哪些认识需要修正，是否需要调整主要矛盾、主要方面或执行顺序。

## 原则

1. 先分析矛盾，再执行任务；没有分析的行动容易变成盲目试错。
2. 先找主要矛盾，再处理次要矛盾；不要把所有问题平均用力。
3. 先抓根因，再处理表象；最响的症状不一定是决定性原因。
4. 先判断阶段，再选择方法；探索期、实现期、故障期、收敛期需要不同策略。
5. 必须具体问题具体分析；通用框架只能指导提问，不能替代事实判断。
6. 内部结构通常决定外部条件如何起作用；外部压力要落实到系统内部机制上分析。
7. 不同矛盾要用不同方法解决；性能、需求、架构、协作、写作结构不能套同一处方。
8. 主要矛盾和矛盾的主要方面会随条件变化而变化；反馈出现后必须重新判断。
9. 执行必须可观察、可验证；每一步都要定义证据和停止条件。
10. 每次实践都要产生反馈；反馈必须反过来修正分析。
11. 避免一次性做过大改动；优先选择能减少不确定性的小步行动。
12. 始终围绕当前主要矛盾推进，次要问题先记录，除非它阻塞主线。
13. 理论必须回到实践；计划、判断和方案只有经过结果检验才算站得住。
14. 实践也必须受分析指导；不要把零散经验误认为完整规律。

## 工作流程

### 第 0 步：明确任务边界

先界定问题，不急于给方案：

- 用户真正要达成什么目标。
- 当前范围内包括什么，不包括什么。
- 必须遵守哪些约束：时间、风险、兼容性、成本、质量、数据、权限、风格或用户偏好。
- 什么结果算完成，什么结果只算局部进展。
- 已知事实有哪些，缺失事实有哪些。

信息不足时，明确假设并标出需要验证的部分，不要伪装确定。

### 第 1 步：识别矛盾

列出塑造任务的冲突关系。常见来源包括：

- 目标冲突：速度与正确性、简单性与扩展性、创新与稳定。
- 系统冲突：接口与实现、数据模型与界面、局部状态与远端事实。
- 过程冲突：范围与时间、探索与交付、个人执行与协作依赖。
- 证据冲突：测试通过但用户失败、日志指向甲但行为像乙、指标与直觉不一致。
- 内外冲突：外部依赖、环境、用户行为、资源限制如何作用于内部代码、流程、结构或假设。

把表象当线索，不要直接当结论。

### 第 2 步：找出主要矛盾

在多个矛盾中找出当前最能规定或影响全局的一个。

判断问题：

- 如果这个矛盾解决，哪些其他问题会变简单或消失。
- 如果这个矛盾不解决，哪些工作会继续失败。
- 哪个矛盾最能解释已经观察到的事实。
- 哪个矛盾最影响下一步行动或决策。

输出时用一句话命名主要矛盾，避免平铺罗列。

### 第 3 步：判断矛盾的主要方面

在主要矛盾内部，判断当前哪一侧起主导作用。主要方面决定问题当前性质，也决定优先行动点。

示例：

- “交付速度与正确性”的矛盾中，生产故障阶段通常由正确性主导，早期原型阶段可能由验证速度主导。
- “前端表现与后端数据契约”的矛盾中，如果所有前端问题都源于异常响应结构，后端契约就是主要方面。
- “宏大范围与有限能力”的矛盾中，如果计划持续延期，团队能力或范围控制往往是主要方面。

主要方面不是永久固定的。条件变化、阶段变化或反馈变化后，要重新判断。

### 第 4 步：判断当前阶段

先判断阶段，再选择行动方式：

- 探索阶段：事实不足，先缩小未知范围。
- 诊断阶段：症状清楚但原因不明，先隔离因果链。
- 设计阶段：冲突和约束已知，先确定结构和取舍。
- 实现阶段：方案清楚，做小范围、可验证的改动。
- 稳定阶段：功能存在，重点验证边界、回归和可维护性。
- 恢复阶段：系统受损，先止血、恢复可控状态。
- 演化阶段：环境或目标变了，要重新判断旧方案是否仍适用。

阶段改变时，主要矛盾可能随之改变。

### 第 5 步：制定实践计划

把分析转化为可执行的小计划：

1. 选定一个直接针对主要矛盾的下一步行动。
2. 说明这个行动要验证哪个判断或减少哪个不确定性。
3. 预先定义期望证据：测试结果、日志、指标、用户反馈、行为变化、文本可读性或方案一致性。
4. 选择最小有效行动：一次复现、一个实验、一个小补丁、一次阅读、一个原型或一次确认。
5. 设置停止条件：什么结果表示继续、修正、回退或放弃该路径。
6. 将次要问题放入暂存列表，除非它直接阻塞当前验证。

### 第 6 步：执行、观察、验证

执行时保持行动和反馈对应：

- 一次只做一个连贯的调查或修改步骤。
- 观察具体反馈，而不是只凭感觉判断。
- 将实际结果与预期证据比较。
- 对意外结果保持敏感，不要强行塞回原来的解释。
- 代码任务优先复现、定位、聚焦修改和测试验证，避免无根据的大重写。

### 第 7 步：修正认识

每个反馈点之后更新分析：

- 原先判断的主要矛盾是否仍成立。
- 矛盾的主要方面是否发生转化。
- 是否有次要矛盾上升为主要矛盾。
- 外部条件是否暴露了内部结构问题。
- 当前阶段是否发生变化。
- 哪些假设被证实、削弱或推翻。

修正认识不是失败，而是方法本身的一部分。

### 第 8 步：迭代推进

重复“分析、实践、反馈、修正”，直到达到成功标准或任务边界发生变化。

每一轮都应该更具体：

- 从表面症状推进到内部结构。
- 从笼统不确定推进到明确因果。
- 从粗略方案推进到验证过的行动。
- 从一次性处理推进到可复用经验。

避免无限分析。只要下一步行动能减少关键不确定性，就进入实践。

### 第 9 步：复盘与固化

任务完成后做轻量复盘：

- 主要矛盾是什么，最终如何解决。
- 哪些反馈改变了原计划。
- 哪些知识应沉淀为测试、文档、规则、提示词、运行手册或示例。
- 仍有哪些次要风险，但不要让它们遮蔽已完成的主要目标。

## 输出模板

复杂任务优先使用以下结构；简单任务可以压缩，但保留“主要矛盾、下一步行动、验证标准”。

```
## 任务边界

- 目标：
- 范围：
- 约束：
- 成功标准：
- 已知事实：
- 缺失事实：

## 矛盾识别

- 主要矛盾：
- 矛盾的主要方面：
- 次要矛盾：
- 内部原因：
- 外部条件：

## 阶段判断

- 当前阶段：
- 判断依据：
- 阶段变化信号：

## 实践计划

1. 下一步行动：
2. 要验证的判断：
3. 预期证据：
4. 验证方法：
5. 停止或修正条件：

## 反馈

- 实际发生了什么：
- 哪些判断被证实：
- 哪些判断被推翻：
- 出现了什么新条件：

## 修正方案

- 更新后的主要矛盾：
- 更新后的主要方面：
- 更新后的下一步行动：
- 剩余风险：
```

## 禁止事项

- 不要把两篇材料写成摘要、读书笔记或哲学散文。
- 不要使用宣传口吻、历史标签或口号替代具体分析。
- 不要把最显眼的症状自动当成根因。
- 不要把所有问题平均看待。
- 不要机械套模板；模板只提供检查顺序，不提供答案。
- 不要脱离事实凭空推理，也不要停留在零散事实不做综合。
- 不要没有验证标准就执行。
- 不要用一次大改动替代一组可反馈的小步行动。
- 不要在反馈推翻判断后继续维护原判断。
- 不要忽视阶段变化；上一阶段的重点可能在下一阶段变成次要问题。
- 不要把所有矛盾都升级成对抗；有些问题需要协调、重构、澄清或渐进修正。
- 不要在结束时省略验证结果或剩余不确定性。

## 示例

### 示例一：代码排错

场景：一个应用同时出现登录失败、接口超时和界面状态错乱。

使用本技能时，不直接把三个症状分别修补，而是先判断它们是否由同一个主要矛盾驱动。主要矛盾可能是“客户端会话假设与服务端令牌过期行为不一致”。如果每个界面错误都跟随异常刷新响应出现，那么服务端数据契约可能是主要方面。实践计划应先复现令牌生命周期，检查一个关键请求链路，补充聚焦测试，再根据结果修正判断。

### 示例二：产品决策

场景：团队在“尽快上线”和“完善新手引导”之间争执。

使用本技能时，先判断产品阶段。如果仍在验证需求，主要矛盾可能是“学习速度与功能完整度”，主要方面可能是学习速度。实践计划应上线最小可用引导实验，预先定义激活率、完成率或留存反馈，再用实际数据修正产品计划。

### 示例三：写作或研究

场景：材料很多，但文章没有清晰论点。

使用本技能时，主要矛盾可能是“大量观察与单一主论点之间的冲突”。实践计划应先写出一句中心判断，再检查每一节是否服务于该判断，删除不能支撑主线的枝节，并让读者复述文章要解决的问题。

### 示例四：人工智能智能体自我校正

场景：智能体已经执行多步，但结果偏离用户目标。

使用本技能时，先停止继续堆动作，重新判断任务边界和主要矛盾。主要矛盾可能是“用户真实目标与智能体自行扩展范围之间的冲突”。实践计划应回到用户原始要求，列出已做动作和偏差，选择最小修正路径，并在下一步前定义验证标准。
